# Local-to-Web Task Transfer Notes

This note summarizes the transfer pattern used to convert a local `psyflow` task into a browser task powered by `psyflow-web`.
It uses `T000006-mid -> H000006-mid` as the reference case, but the process is meant to be generic and reusable for other tasks.

## Core Principle

Do not try to make the browser task a line-by-line translation of the local Python task.
Instead, keep the same task contract and authoring structure, then re-express the execution in `psyflow-web`.

What should stay aligned:

- task identity: same `slug`, same conceptual title, `T` for local and `H` for html
- task structure: `main`, `config`, `run_trial`, `controller`, `utils`
- trial procedure: same stage order and response logic
- parameters: same timing, controller logic, instructions, scoring, and condition semantics whenever possible
- data meaning: exported trial rows should preserve the same analysis logic

What can differ:

- hardware-specific layers such as EEG triggers
- deployment/runtime layer
- browser-specific UI details such as fullscreen, cursor hiding, and force-quit behavior
- preview-length settings such as fewer blocks or trials

## Recommended Folder Shape

For the html task repo, keep a TAPS-like layout instead of a generic frontend app layout:

```text
Hxxxxxx-task/
  main.ts
  README.md
  taskbeacon.yaml
  config/
    config.yaml
  src/
    run_trial.ts
    controller.ts
    utils.ts
```

The task repo should contain task source only.
Do not put per-task `node_modules`, `vite.config`, or app-shell boilerplate here.
Those belong in `psyflow-web`.

## Mapping Local Task Concepts to Web Task Concepts

Local `psyflow` concept -> `psyflow-web` concept

- `main.py` -> `main.ts`
- `config/config.yaml` -> `config/config.yaml`
- `src/run_trial.py` -> `src/run_trial.ts`
- `StimUnit(...).show/capture_response/wait_and_continue` -> `trial.unit(...).show/captureResponse/waitAndContinue`
- `BlockUnit.generate_conditions()` -> same conceptual use in web
- `StimBank` -> same concept, but rendered by browser primitives
- `controller` -> same task-local timing/adaptation helper
- `SubInfo` form -> browser pre-task form generated by `TaskApp`
- local runtime/export -> `mountTaskApp(...)` plus raw/reduced export

## Migration Workflow

### 1. Choose a canonical local task

Treat the local `Txxxxxx-*` repo as the source of truth.
The html version should align to that task, not diverge into a separate implementation.

At minimum, extract from the local task:

- procedure order
- condition set
- timing parameters
- controller rules
- instruction text
- score/outcome logic
- task metadata (`slug`, title, acquisition, maturity)

### 2. Create the html task identity

Create a matching `Hxxxxxx-*` repo with the same `slug`.

Example:

- local: `T000006-mid`
- web: `H000006-mid`

In `taskbeacon.yaml`:

- keep the same `slug`
- set `variant: html`
- use an `H` id matching the local task number when possible

This makes gallery pairing automatic.

### 3. Keep `main` orchestration explicit

Do not hide the task flow behind a heavy abstraction layer.
`main.ts` should remain readable in the same way `main.py` is readable.

Typical responsibilities of `main.ts`:

- load and parse config
- create `TaskSettings`, `SubInfo`, `StimBank`, and `controller`
- build instruction screens
- generate blocks and conditions
- call `run_trial(...)` for each logical trial
- insert generic screens such as countdowns and block breaks
- hand everything to `mountTaskApp(...)`

This preserves task-level auditability.

### 4. Rewrite `run_trial` around trial builders, not browser globals

The seam between authoring and runtime should be at `run_trial`.

Local pattern:

- direct use of `win`, `kb`, trigger runtime, and immediate execution

Web pattern:

- build a logical trial with `TrialBuilder`
- declare stages with `unit(...).show/captureResponse/waitAndContinue`
- pass task services in a context object:
  - `settings`
  - `stimBank`
  - `controller`
  - `utils`

Target shape:

```ts
run_trial(trial, condition, { settings, stimBank, controller, utils })
```

The web author should not import `jsPsych` directly.

### 5. Preserve trial semantics before optimizing UX

Port the logical task first.
Only after the browser task matches the local task should you add web-specific behavior.

Priority order:

1. stage order
2. response windows
3. correctness/hit logic
4. feedback logic
5. controller updates
6. instructions and voice
7. fullscreen/cursor/quit behavior

For preview builds, shorten counts only after the single-trial behavior is aligned.

Good preview differences:

- fewer blocks
- fewer trials

Bad preview differences:

- different controller rules
- different timing logic
- different score semantics

### 6. Move generic browser concerns into `psyflow-web`

If a behavior is not task-specific, do not re-implement it in each `H` repo.

Examples that belong in `psyflow-web`:

- fullscreen entry and guarding
- cursor hiding for keyboard tasks
- generic preflight/start screen
- `SubInfo` form rendering
- export UI
- force-quit shortcut
- generic countdown helper
- browser speech synthesis bridge

Task repos should only describe task logic and task-specific text/assets.

### 7. Keep stimuli config-driven

Keep participant-facing content in config/stimuli whenever possible.

That includes:

- instruction text
- cue/feedback text
- image references
- sound references
- formatting templates

This keeps `run_trial.ts` focused on procedure instead of content storage.

### 8. Treat controller and utils as portable task logic

Most task-specific pure logic should move with minimal change.

Good candidates for near-direct transfer:

- adaptive response-window logic
- outcome computation
- block summary helpers
- condition formatting helpers

If logic is pure and independent from PsychoPy objects, keep it near-identical.

### 9. Validate the data contract

The browser runtime may log more granular stage data internally, but exported task data should stay analysis-friendly.

Recommended rule:

- raw data: keep stage-level audit records
- reduced data: one logical trial per row

When transferring a task, check that:

- trial ids are stable
- block ids/condition ids are preserved
- response and RT fields still mean the same thing
- feedback and score columns remain analyzable by the same logic

### 10. Deploy with a shared runner

The html task repo is source.
The runnable browser app is served by `psyflow-web`.

Recommended deployment model:

- deploy `psyflow-web` to GitHub Pages
- let the shared runner fetch or bundle public `H` task repos
- point gallery run links to:
  - `https://taskbeacon.github.io/psyflow-web/?task=Hxxxxxx-task`

This avoids building a separate frontend app for every task.

## Checklist for Porting Another Task

- create `Hxxxxxx-*` repo with matching `slug`
- copy the TAPS-style folder structure
- port `config` first
- port `main` orchestration without over-abstracting it
- port `run_trial` to the trial-builder pattern
- port pure `controller` and `utils` logic
- align stage order and response semantics with the local task
- keep preview-only changes limited to task length
- move generic browser concerns into `psyflow-web`
- verify raw and reduced exports
- register metadata so the gallery can pair the local and html variants
- deploy through the shared `psyflow-web` runner

## Short Rule

The local task remains the canonical experiment.
The html task is a web-native re-authoring of the same contract.
`psyflow-web` should absorb runtime and browser complexity so individual task repos stay small, auditable, and close to their local `psyflow` counterparts.
